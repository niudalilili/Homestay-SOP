 SaToken RBAC改造计划

 上下文

 当前项目使用SaToken进行权限控制，但实现存在以下问题：

 当前SaToken实现的问题分析

 1. 非标准RBAC实现：当前权限模型是简单的单角色字符串存储，缺乏标准的RBAC（基于角色的访问控制）要素：
   - 没有角色表、权限表、菜单表等实体定义
   - 角色直接存储在employee表的role字段中，为简单的字符串类型
   - 缺乏角色-权限、用户-角色、角色-菜单等多对多关系
 2. 权限控制不完整：
   - 只有EmployeeController使用@SaCheckRole("admin")注解
   - ActivityPlanController中所有接口公开（仅受全局登录拦截器保护）
   - 用户端（user）框架预留但未实现任何权限控制
 3. 权限粒度不足：
   - StpInterfaceImpl.getPermissionList()返回空列表，未实现细粒度权限控制
   - 只有角色级别的控制，缺乏权限级别的控制
 4. 硬编码逻辑：
   - StpInterfaceImpl中硬编码了admin账号的特殊处理
   - 角色管理通过员工管理界面直接设置字符串值，缺乏规范性
 5. 配置混乱：
   - SaTokenConfigure配置类中的拦截器被注释掉，实际生效的拦截器在WebMvcConfiguration中
   - 配置分散，缺乏统一管理
 6. 缺乏权限管理功能：
   - 没有角色管理、权限管理、菜单管理接口
   - 无法动态分配权限和角色

 RBAC标准要求

 标准的RBAC（基于角色的访问控制）应包含以下核心要素：

 1. 用户（User）：系统的使用者
 2. 角色（Role）：权限的集合
 3. 权限（Permission）：具体的操作权限
 4. 资源/菜单（Menu）：系统功能模块
 5. 关系映射：用户-角色、角色-权限、权限-菜单等多对多关系

 项目现状

 - 技术栈：Spring Boot 3 + SaToken 1.44.0 + MyBatis
 - 当前权限模型：简单的单角色字符串模型
 - 数据库：只有employee表包含role字段，无其他权限相关表
 - 代码基础：已集成SaToken框架，StpInterfaceImpl预留了权限扩展点

 用户决策确认

 1. RBAC级别：基础RBAC0（用户-角色-权限基础模型）
 2. 数据迁移：全新开始（不迁移现有role字段数据）
 3. 前端权限控制：只需后端API权限（前端暂不集成）
 4. 权限编码规范：模块:操作格式（如employee:add）
 5. 默认角色：预定义管理员(admin)和普通员工(employee)角色
 6. 菜单管理：暂不需要（只实现接口权限控制）
 7. Admin特殊逻辑：完全RBAC化（取消硬编码，通过角色权限表管理）
 8. Employee.role字段：立即移除（完全使用user_role关联表）

 实施计划

 实施顺序说明

 1. 阶段一（数据库与实体）：必须最先完成，为后续提供数据基础
 2. 阶段二（服务层）：依赖阶段一完成，实现核心业务逻辑
 3. 阶段三（控制器层）：依赖阶段二完成，提供API接口
 4. 阶段四（接口层）：可并行开发，提供基础权限查询
 5. 阶段五（测试验证）：贯穿整个开发过程

 技术细节

 SaToken权限注解使用

 // 角色检查（不推荐，使用权限检查更灵活）
 @SaCheckRole("admin")

 // 权限检查（推荐）
 @SaCheckPermission("employee:add")
 @SaCheckPermission("employee:query")
 @SaCheckPermission("activity:add")

 权限编码规范设计

 编码格式：模块:操作

 - 模块：表示功能模块，如employee、activity、system等
 - 操作：表示具体操作，如add、update、delete、query、status等

 具体权限编码列表

 员工管理模块 (employee)
 - employee:add - 新增员工（对应save方法）
 - employee:query - 查询员工（对应page、getById方法）
 - employee:update - 编辑员工信息（对应update方法）
 - employee:status - 启用禁用员工账号（对应startOrStop方法）
 - employee:password - 编辑员工密码（对应editPassword方法）
 - employee:delete - 删除员工（对应delete方法）

 活动管理模块 (activity)
 - activity:add - 新增活动方案（对应save方法）
 - activity:query - 查询活动方案（对应page、selectById、stats方法）
 - activity:update - 更新活动方案（对应update方法）
 - activity:status - 修改活动状态（对应changeStatus方法）
 - activity:delete - 删除活动方案（对应deleteById方法）

 系统管理模块 (system)
 - system:role - 角色管理
 - system:permission - 权限管理
 - system:userRole - 用户角色分配

 权限注解使用示例

 // EmployeeController
 @SaCheckPermission("employee:add")
 public Result save(@RequestBody EmployeeDTO employeeDTO)

 @SaCheckPermission("employee:query")
 public Result<PageResult> page(EmployeePageQueryDTO employeePageQueryDTO)

 // ActivityPlanController
 @SaCheckPermission("activity:add")
 public Result save(@RequestBody ActivityPlanDTO activityPlanDTO)

 @SaCheckPermission("activity:query")
 public Result<PageResult> page(ActivityPlanPageQueryDTO activityPlanPageQueryDTO)

 缓存策略

 - 用户权限列表缓存：用户登录时查询权限并缓存，减少每次鉴权的数据库查询
 - 角色权限关系缓存：缓存角色-权限映射关系，提高权限验证速度
 - 使用SaToken自带的缓存机制，可配置缓存时间
 - 考虑缓存失效策略：权限变更时清除相关缓存

 风险与缓解

 风险1：数据库迁移数据丢失

 - 缓解：备份现有数据，提供回滚脚本
 - 测试环境充分验证迁移脚本

 风险2：权限控制过严导致现有功能不可用

 - 缓解：分阶段实施，先添加权限注解但不启用严格检查
 - 提供权限开关配置

 风险3：性能问题

 - 缓解：实现合理的缓存策略
 - 优化数据库查询，使用索引

 风险4：前端兼容性问题

 - 缓解：提供兼容接口，逐步过渡
 - 前端权限控制作为可选阶段
